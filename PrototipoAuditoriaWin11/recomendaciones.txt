#PasswordHistorySize
Descripción:

Esta configuración de política determina cuántas contraseñas renovadas y únicas deben asociarse con una cuenta de usuario antes de poder reutilizar una contraseña antigua. El valor debe estar entre 0 y 24 contraseñas. El valor predeterminado para sistemas independientes es 0 contraseñas, mientras que para sistemas unidos a un dominio es 24 contraseñas. Para mantener la efectividad de esta política, se recomienda usar la configuración de Edad mínima de la contraseña para evitar cambios repetidos. Estado recomendado: 24 o más contraseñas.

Nota:

Las configuraciones de Política de contraseñas y de Política de bloqueo de cuentas deben aplicarse mediante la GPO de Política de dominio predeterminada para tener efecto globalmente en cuentas de usuario de dominio. Las excepciones a estas reglas pueden definirse usando Objetos de configuración de contraseña, independientes de la Política de grupo.

Nota #2:

Microsoft tiene un límite máximo de 24 contraseñas guardadas. Para más información, visita la documentación de Microsoft sobre Imponer el historial de contraseñas en Windows 10.

Justificación:

El uso prolongado de una misma contraseña aumenta el riesgo de ataques de fuerza bruta y explotación de cuentas comprometidas. Evitar la reutilización de contraseñas y establecer una edad mínima adecuada ayuda a mantener la seguridad. Un valor bajo en esta configuración permite la reutilización frecuente de contraseñas, reduciendo la efectividad de la política de contraseñas.

Impacto:

Esta configuración obliga a los usuarios a crear nuevas contraseñas en cada cambio, lo que puede llevar a prácticas de seguridad inadecuadas, como anotar las contraseñas. También aumenta el riesgo de contraseñas predecibles. Un valor bajo en la edad mínima de la contraseña puede aumentar la carga administrativa al incrementar los reinicios de contraseña solicitados por los usuarios.

#MaximumPasswordAge
Descripción:

Esta configuración de política define cuánto tiempo puede usar un usuario su contraseña antes de que expire. Los valores van de 0 a 999 días. Si establece el valor en 0, la contraseña nunca caducará. Aunque los atacantes pueden descifrar contraseñas, cuanto más frecuentemente cambie la contraseña, menos oportunidades tendrán los atacantes de usar una contraseña descifrada. Sin embargo, cuanto más bajo esté configurado este valor, mayor será el potencial de aumento en las llamadas al soporte técnico debido a que los usuarios tienen que cambiar su contraseña o olvidar cuál es la contraseña actual. El estado recomendado para esta configuración es de 365 días o menos, pero no 0.

Nota:

Las configuraciones de Política de contraseñas y de Política de bloqueo de cuentas deben aplicarse mediante la GPO de Política de dominio predeterminada para tener efecto globalmente en cuentas de usuario de dominio. Las excepciones a estas reglas pueden definirse usando Objetos de configuración de contraseña, independientes de la Política de grupo.

Justificación:

Cuanto más tiempo exista una contraseña, mayor será la probabilidad de que sea comprometida por un ataque de fuerza bruta, por un atacante que obtenga conocimientos generales sobre el usuario o por el usuario que comparta la contraseña. Configurar la configuración de Edad máxima de la contraseña en 0 para que los usuarios nunca estén obligados a cambiar sus contraseñas es un riesgo de seguridad importante, ya que permite que una contraseña comprometida sea utilizada por el usuario malintencionado durante el tiempo que el usuario válido tenga acceso autorizado.

Impacto:

Si la configuración de Edad máxima de la contraseña es demasiado baja, se requiere que los usuarios cambien sus contraseñas con mucha frecuencia. Esta configuración puede reducir la seguridad en la organización, ya que los usuarios podrían escribir sus contraseñas en un lugar inseguro o perderlas. Si el valor para esta configuración de política es demasiado alto, se reduce el nivel de seguridad dentro de una organización porque permite a los posibles atacantes más tiempo para descubrir contraseñas de usuarios o para usar cuentas comprometidas.

#MinimumPasswordAge
Descripción:

Esta configuración de política determina el número de días que debe usar una contraseña antes de poder cambiarla. El rango de valores para esta configuración de política es entre 1 y 999 días. (También puede establecer el valor en 0 para permitir cambios de contraseña inmediatos). El valor predeterminado para esta configuración es de 0 días. El estado recomendado para esta configuración es: 1 o más día(s).

Nota:

Las configuraciones de Política de contraseñas y de Política de bloqueo de cuentas deben aplicarse mediante la GPO de Política de dominio predeterminada para tener efecto globalmente en cuentas de usuario de dominio. Las excepciones a estas reglas pueden definirse usando Objetos de configuración de contraseña, independientes de la Política de grupo.

Justificación:

Los usuarios pueden tener contraseñas favoritas que les guste usar porque son fáciles de recordar y creen que su elección de contraseña es segura contra compromisos. Desafortunadamente, las contraseñas son comprometidas y si un atacante está apuntando a la cuenta de usuario específica de un individuo, con conocimiento previo de datos sobre ese usuario, la reutilización de contraseñas antiguas puede causar una brecha de seguridad. Para abordar la reutilización de contraseñas se requiere una combinación de configuraciones de seguridad. El uso de esta configuración de política con la configuración de Imponer el historial de contraseñas evita la fácil reutilización de contraseñas antiguas. Por ejemplo, si configura la configuración de Imponer el historial de contraseñas para asegurarse de que los usuarios no puedan reutilizar ninguna de sus últimas 12 contraseñas, podrían cambiar su contraseña 13 veces en pocos minutos y reutilizar la contraseña con la que comenzaron, a menos que también configure la configuración de Edad mínima de la contraseña a un número mayor que 0. Debe configurar esta configuración de política a un número mayor que 0 para que la configuración de Imponer el historial de contraseñas sea efectiva.

Impacto:

Si un administrador establece una contraseña para un usuario pero quiere que ese usuario cambie la contraseña cuando inicie sesión por primera vez, el administrador debe seleccionar la casilla de verificación El usuario debe cambiar la contraseña en el próximo inicio de sesión, o el usuario no podrá cambiar la contraseña hasta el próximo día.

#MinimumPasswordLength
Descripción:

Esta configuración de política determina el número mínimo de caracteres que componen una contraseña para una cuenta de usuario. Existen muchas teorías diferentes sobre cómo determinar la mejor longitud de contraseña para una organización, pero quizás "frase de contraseña" sea un término más adecuado que "contraseña". En Microsoft Windows 2000 y versiones más recientes, las frases de contraseña pueden ser bastante largas e incluir espacios. Por lo tanto, una frase como "Quiero tomar un batido de $5" es una frase de contraseña válida; es una contraseña considerablemente más fuerte que una cadena de 8 o 10 caracteres de números y letras aleatorios, y sin embargo, es más fácil de recordar. Los usuarios deben ser educados sobre la selección y el mantenimiento adecuados de las contraseñas, especialmente en lo que respecta a la longitud de las contraseñas. En entornos empresariales, el valor ideal para la configuración de Longitud mínima de la contraseña es de 14 caracteres, sin embargo, debe ajustar este valor para cumplir con los requisitos comerciales de su organización. El estado recomendado para esta configuración es: 14 o más caracteres.

Nota:

En Windows Server 2016 y versiones anteriores de Windows Server, la interfaz gráfica de la Política de seguridad local (LSP), el Editor de directivas de grupo local (LGPE) y el Editor de administración de directivas de grupo (GPME) no permitían establecer este valor en más de 14 caracteres. Sin embargo, a partir de Windows Server 2019, Microsoft cambió la interfaz gráfica para permitir una longitud mínima de contraseña de hasta 20 caracteres.

Nota #2:

Las configuraciones de Política de contraseñas y de Política de bloqueo de cuentas deben aplicarse mediante la GPO de Política de dominio predeterminada para tener efecto globalmente en cuentas de usuario de dominio. Las excepciones a estas reglas pueden definirse usando Objetos de configuración de contraseña, independientes de la Política de grupo.

Justificación:

Los tipos de ataques de contraseña incluyen ataques de diccionario (que intentan usar palabras y frases comunes) y ataques de fuerza bruta (que prueban todas las combinaciones posibles de caracteres). Además, los atacantes a veces intentan obtener la base de datos de cuentas para usar herramientas y descubrir las cuentas y contraseñas.

Impacto:

Los requisitos de contraseñas extremadamente largas pueden, en realidad, disminuir la seguridad de una organización, porque los usuarios podrían dejar la información en un lugar inseguro o perderla. Si se requieren contraseñas muy largas, las contraseñas mal escritas podrían causar bloqueos de cuentas y aumentar el volumen de llamadas al servicio de asistencia. Si su organización tiene problemas con contraseñas olvidadas debido a los requisitos de longitud de contraseña, considere enseñar a sus usuarios sobre las frases de contraseña, que son a menudo más fáciles de recordar y, debido al mayor número de combinaciones de caracteres, mucho más difíciles de descubrir.

Nota: Las versiones más antiguas de Windows, como Windows 98 y Windows NT 4.0, no admiten contraseñas de más de 14 caracteres. Las computadoras que ejecutan estos sistemas operativos antiguos no pueden autenticarse con computadoras o dominios que usen cuentas que requieran contraseñas largas.

#PasswordComplexity
Descripción:
Esta configuración de política verifica todas las nuevas contraseñas para asegurar que cumplan con los requisitos básicos de contraseñas seguras. Cuando esta política está habilitada, las contraseñas deben cumplir los siguientes requisitos mínimos:

- No contener el nombre de la cuenta del usuario ni partes del nombre completo del usuario que excedan dos caracteres consecutivos.
- Tener al menos seis caracteres de longitud.
- Contener caracteres de tres de las siguientes categorías:
- Caracteres en mayúsculas en inglés (A a Z).
- Caracteres en minúsculas en inglés (a a z).
- Dígitos del 0 al 9.
- Caracteres no alfabéticos (por ejemplo, !, $, #, %).
- Una categoría general de cualquier carácter Unicode que no pertenezca a las cuatro categorías anteriores, pudiendo ser específicos de la región.
- Cada carácter adicional en una contraseña aumenta su complejidad exponencialmente. El estado recomendado para esta configuración es: Habilitado.

Nota:
Las configuraciones de Política de contraseñas y de Política de bloqueo de cuentas deben aplicarse mediante la GPO de Política de dominio predeterminada para tener efecto global en cuentas de usuario de dominio. Las excepciones a estas reglas pueden definirse usando Objetos de configuración de contraseña, independientes de la Política de grupo.

Justificación:
Las contraseñas que contienen solo caracteres alfanuméricos son extremadamente fáciles de descubrir con varias herramientas disponibles públicamente.

Impacto:
Si se mantiene la configuración de complejidad de contraseñas predeterminada, podrían aumentar las llamadas al servicio de asistencia debido a cuentas bloqueadas, ya que los usuarios podrían no estar acostumbrados a contraseñas que contengan caracteres no alfabéticos. Sin embargo, todos los usuarios deberían poder cumplir con el requisito de complejidad con mínima dificultad.

Para organizaciones con requisitos de seguridad más estrictos, se puede crear una versión personalizada del archivo Passfilt.dll para permitir reglas de fortaleza de contraseñas arbitrariamente complejas, como el uso de caracteres no estándar. También se puede requerir el uso de combinaciones de caracteres ALT para aumentar significativamente la complejidad de una contraseña. Sin embargo, requisitos de contraseñas tan estrictos pueden resultar en usuarios descontentos y un servicio de asistencia muy ocupado.

Alternativamente, se podría requerir que todas las contraseñas de administradores utilicen caracteres ALT en el rango 0128 - 0159.

#MACHINE\\System\\CurrentControlSet\\Control\\SAM\\RelaxMinimumPasswordLengthLimits
Descripción:
Esta configuración de política determina si la longitud mínima de la contraseña puede aumentarse más allá del límite heredado de 14 caracteres. Estado recomendado: Habilitado.

Nota:
Esta configuración solo afecta a las cuentas locales en el equipo. Las cuentas de dominio solo se ven afectadas por las configuraciones en los Controladores de Dominio, ya que es allí donde se almacenan las cuentas de dominio.

Justificación:
Esta configuración permitirá la aplicación de contraseñas o frases de paso más largas y, generalmente, más seguras donde no se utilice MFA (Autenticación Multifactor).

Impacto:
La configuración de la longitud mínima de la contraseña puede configurarse para ser mayor a 14 caracteres. Si se requieren contraseñas muy largas, las contraseñas mal escritas podrían causar bloqueos de cuenta y aumentar el volumen de llamadas al servicio de asistencia. Si tu organización tiene problemas con contraseñas olvidadas debido a los requisitos de longitud, considera enseñar a tus usuarios sobre las frases de paso, que a menudo son más fáciles de recordar y, debido al mayor número de combinaciones de caracteres, mucho más difíciles de descubrir.

#ClearTextPassword
Descripción:
Esta configuración de política determina si el sistema operativo almacena contraseñas de manera que utilice cifrado reversible, lo que proporciona soporte para protocolos de aplicación que requieren conocimiento de la contraseña del usuario para propósitos de autenticación. Las contraseñas almacenadas con cifrado reversible son esencialmente iguales a versiones en texto plano de las contraseñas. Estado recomendado: Deshabilitado.

Nota:
Las configuraciones de Política de contraseñas y de Política de bloqueo de cuentas deben aplicarse mediante la GPO de Política de dominio predeterminada para tener efecto global en cuentas de usuario de dominio. Las excepciones a estas reglas pueden definirse usando Objetos de configuración de contraseña, independientes de la Política de grupo.

Justificación:
Habilitar esta configuración permite que el sistema operativo almacene contraseñas en un formato más débil, lo que es mucho más susceptible a compromisos y debilita la seguridad del sistema.

Impacto:
Si tu organización usa el protocolo de autenticación CHAP a través de acceso remoto o servicios IAS o Autenticación Digest en IIS, debes configurar esta política como Habilitada. Esta configuración es extremadamente peligrosa de aplicar a través de la Política de grupo en una base de usuario por usuario, ya que requiere abrir el objeto de cuenta de usuario apropiado en Usuarios y equipos de Active Directory.

#LockoutDuration
Descripción:
Esta configuración de política determina el tiempo que debe pasar antes de que una cuenta bloqueada se desbloquee y el usuario pueda intentar iniciar sesión nuevamente. Especifica el número de minutos que una cuenta bloqueada permanecerá inaccesible. Si el valor de esta configuración se establece en 0, las cuentas bloqueadas permanecerán bloqueadas hasta que un administrador las desbloquee manualmente. Aunque podría parecer una buena idea configurar un valor alto, esto probablemente aumentará el número de llamadas al servicio de asistencia para desbloquear cuentas bloqueadas por error. Estado recomendado: 15 o más minutos.

Nota:
Las configuraciones de Política de contraseñas y de Política de bloqueo de cuentas deben aplicarse mediante la GPO de Política de dominio predeterminada para tener efecto global en cuentas de usuario de dominio. Las excepciones a estas reglas pueden definirse usando Objetos de configuración de contraseña, independientes de la Política de grupo.

Justificación:
Se puede crear una condición de denegación de servicio (DoS) si un atacante abusa del umbral de bloqueo de cuenta e intenta repetidamente iniciar sesión con una cuenta específica. Una vez que configuras el umbral de bloqueo de cuenta, la cuenta se bloqueará después del número especificado de intentos fallidos. Si configuras la duración del bloqueo de cuenta en 0, la cuenta permanecerá bloqueada hasta que un administrador la desbloquee manualmente.

Impacto:
Aunque puede parecer una buena idea configurar esta política para que nunca desbloquee automáticamente una cuenta, esto puede aumentar el número de solicitudes que recibe el servicio de asistencia para desbloquear cuentas que fueron bloqueadas por error.

#LockoutBadCount
Descripción:

Esta configuración de política determina el número de intentos de inicio de sesión fallidos antes de que se bloquee la cuenta. Establecer esta política en 0 no cumple con el estándar, ya que desactiva el umbral de bloqueo de cuenta. El estado recomendado para esta configuración es: 5 o menos intentos de inicio de sesión inválidos, pero no 0.

Nota:

Las configuraciones de Política de contraseñas y de Política de bloqueo de cuentas deben aplicarse mediante la GPO de Política de dominio predeterminada para tener efecto globalmente en cuentas de usuario de dominio. Sin embargo, se pueden definir excepciones personalizadas a las reglas predeterminadas utilizando Objetos de configuración de contraseña.

Justificación:

Establecer un umbral de bloqueo de cuenta reduce la probabilidad de éxito de un ataque de fuerza bruta en línea. Sin embargo, establecer el umbral de bloqueo de cuenta demasiado bajo aumenta el riesgo de bloqueos accidentales o de un actor malicioso que bloquee intencionalmente cuentas.

Impacto:

Si esta configuración de política está habilitada, una cuenta bloqueada no será utilizable hasta que sea restablecida por un administrador o hasta que expire la duración del bloqueo de cuenta. Esto puede generar llamadas adicionales al servicio de asistencia. Si se aplica esta configuración, un atacante podría causar una denegación de servicio al generar deliberadamente intentos de inicio de sesión fallidos para múltiples usuarios. Por lo tanto, también se debe configurar la Duración del bloqueo de cuenta con un valor relativamente bajo. Si se configura el Umbral de bloqueo de cuenta en 0, existe la posibilidad de que un intento de descubrir contraseñas mediante un ataque de fuerza bruta pase desapercibido si no se cuenta con un mecanismo de auditoría robusto.

#AllowAdministratorLockout
Descripción:

Esta configuración de política determina si la cuenta de Administrador incorporada está sujeta a las siguientes configuraciones de Política de bloqueo de cuentas: Duración del bloqueo de cuenta, Umbral de bloqueo de cuenta y Reinicio del contador de bloqueo de cuenta. Por defecto, esta cuenta está excluida de los controles de bloqueo de cuenta y nunca será bloqueada con intentos repetidos de contraseña incorrecta.

Estado recomendado: Habilitado.

Nota:

Esta configuración se aplica solo a sistemas operativos actualizados hasta el 11 de octubre de 2022 (ver MS KB5020282).

Justificación:

Habilitar las políticas de bloqueo de cuenta para la cuenta de Administrador incorporada reducirá la probabilidad de un ataque de fuerza bruta exitoso.

Impacto:

La cuenta de Administrador incorporada estará sujeta a las políticas en la Sección 1.2 Política de bloqueo de cuentas de este estándar.

#ResetLockoutCount
Descripción:
Este ajuste de política determina el tiempo antes de que el umbral de bloqueo de cuenta se restablezca a cero. Se recomienda establecer este valor en 15 minutos o más.

Justificación:
Para evitar bloqueos accidentales de cuentas debido a errores de contraseña, es crucial definir un tiempo de reinicio para el contador de bloqueo de cuentas después de intentos fallidos de inicio de sesión.

Impacto:
Una configuración inadecuada podría exponer al sistema a ataques de denegación de servicio (DoS), permitiendo que un atacante bloquee cuentas mediante múltiples intentos de inicio de sesión fallidos. Configurar este ajuste adecuadamente garantiza que las cuentas se desbloqueen automáticamente después de un período razonable, reduciendo las llamadas de soporte.

#SeTrustedCredManAccessPrivilege
Descripción:

Este ajuste de seguridad es utilizado por el Administrador de Credenciales durante la Copia de Seguridad y Restauración. Ninguna cuenta debería tener este derecho de usuario, ya que solo se asigna a Winlogon. Las credenciales guardadas de los usuarios podrían estar comprometidas si este derecho de usuario se asigna a otras entidades. El estado recomendado para esta configuración es: Nadie.

Justificación:

Si una cuenta recibe este derecho, el usuario de la cuenta podría crear una aplicación que invoque al Administrador de Credenciales y devuelva las credenciales de otro usuario.

Impacto:

Ninguno, este es el comportamiento predeterminado.

#SeNetworkLogonRight
Descripción:
Este ajuste de política permite a otros usuarios en la red conectarse al equipo y es necesario para varios protocolos de red, incluidos los basados en Server Message Block (SMB), NetBIOS, Common Internet File System (CIFS) y Component Object Model Plus (COM+). La configuración recomendada para este ajuste es: Administradores, Usuarios de Escritorio Remoto.

Justificación:
Los usuarios que pueden conectarse desde su computadora a la red pueden acceder a recursos en otros equipos para los cuales tienen permiso. Por ejemplo, el derecho de usuario "Acceder a este equipo desde la red" es necesario para que los usuarios se conecten a impresoras y carpetas compartidas. Si este derecho se asigna al grupo "Todos", cualquiera podrá leer los archivos en esas carpetas compartidas.

Impacto:
Si eliminas el derecho de usuario "Acceder a este equipo desde la red" en Controladores de Dominio para todos los usuarios, nadie podrá iniciar sesión en el dominio ni usar los recursos de red. Si eliminas este derecho de usuario en Servidores Miembro, los usuarios no podrán conectarse a esos servidores a través de la red. La negociación exitosa de conexiones IPsec requiere que la máquina iniciadora tenga este derecho, por lo que si usas IPsec, se recomienda asignarlo al grupo "Usuarios Autenticados". Si has instalado componentes opcionales como ASP.NET o Servicios de Información de Internet (IIS), es posible que necesites asignar este derecho de usuario a cuentas adicionales requeridas por esos componentes. Es importante verificar que los usuarios autorizados tengan este derecho de usuario para las computadoras a las que necesitan acceder en la red.

#SeTcbPrivilege
Descripción:
Este ajuste de política permite a un proceso asumir la identidad de cualquier usuario y, por lo tanto, obtener acceso a los recursos a los que el usuario está autorizado a acceder. La configuración recomendada para este ajuste es: Nadie.

Justificación:
El derecho de usuario "Actuar como parte del sistema operativo" es extremadamente poderoso. Cualquier persona con este derecho de usuario puede tomar el control completo de la computadora y borrar evidencia de sus actividades.

Impacto:
Debería haber poco o ningún impacto porque el derecho de usuario "Actuar como parte del sistema operativo" rara vez es necesario para cuentas que no sean la cuenta del Sistema Local, que implícitamente tiene este derecho.

#SeIncreaseQuotaPrivilege
Descripción:
Este ajuste de política permite a un usuario ajustar la cantidad máxima de memoria disponible para un proceso. Si bien la capacidad de ajustar cuotas de memoria es útil para la optimización del sistema, también puede ser abusada. En manos equivocadas, podría utilizarse para lanzar un ataque de denegación de servicio (DoS). El estado recomendado para este ajuste es: Administradores, SERVICIO LOCAL, SERVICIO DE RED.

Justificación:
Un usuario con el derecho de usuario "Ajustar cuotas de memoria para un proceso" puede reducir la cantidad de memoria disponible para cualquier proceso, lo que podría hacer que las aplicaciones críticas para el negocio en la red se vuelvan lentas o fallen. En manos equivocadas, este privilegio podría utilizarse para iniciar un ataque de denegación de servicio (DoS).

Impacto:
Las organizaciones que no han restringido a los usuarios a roles con privilegios limitados encontrarán difícil imponer esta contramedida. Además, si ha instalado componentes opcionales como ASP.NET o IIS, es posible que necesite asignar el derecho de usuario "Ajustar cuotas de memoria para un proceso" a cuentas adicionales requeridas por esos componentes. De lo contrario, esta contramedida no debería tener impacto en la mayoría de las computadoras. Si este derecho de usuario es necesario para una cuenta de usuario, se puede asignar a una cuenta de equipo local en lugar de a una cuenta de dominio.

#SelongeractiveLogonRight
Descripción:
Este ajuste de política determina qué usuarios pueden iniciar sesión interactivamente en las computadoras de su entorno. Los inicios de sesión que se inician presionando la secuencia de teclas CTRL+ALT+SUPR en el teclado de la computadora cliente requieren este derecho de usuario. Los usuarios que intentan iniciar sesión a través de Terminal Services / Remote Desktop Services o IIS también requieren este derecho de usuario. El estado recomendado para este ajuste es: Administradores, Usuarios.

Justificación:
Cualquier cuenta con el derecho de usuario "Permitir inicio de sesión localmente" puede iniciar sesión en la consola de la computadora. Si no restringe este derecho de usuario a usuarios legítimos que necesiten poder iniciar sesión en la consola de la computadora, los usuarios no autorizados podrían descargar y ejecutar software malicioso para elevar sus privilegios.

Impacto:
Si elimina estos grupos predeterminados, podría limitar las capacidades de los usuarios asignados a roles administrativos específicos en su entorno. Debe confirmar que las actividades delegadas no se verán afectadas negativamente por los cambios que realice en el derecho de usuario "Permitir inicio de sesión localmente".

#SeRemoteInteractiveLogonRight
Descripción:
Este ajuste de política determina qué usuarios o grupos tienen el derecho de iniciar sesión como cliente de Remote Desktop Services. Si su organización utiliza Asistencia remota como parte de su estrategia de mesa de ayuda, cree un grupo y asígnele este derecho de usuario a través de la Política de grupo. Si la mesa de ayuda en su organización no utiliza Asistencia remota, asigne este derecho de usuario solo al grupo Administradores o use la función de Grupos restringidos para asegurarse de que ninguna cuenta de usuario forme parte del grupo Remote Desktop Users. Restrinja este derecho de usuario al grupo Administradores, y posiblemente al grupo Remote Desktop Users, para evitar que usuarios no deseados obtengan acceso a las computadoras en su red mediante la función de Asistencia remota.

Justificación:
Cualquier cuenta con el derecho de usuario "Permitir inicio de sesión a través de Remote Desktop Services" puede iniciar sesión en la consola remota de la computadora. Si no restringe este derecho de usuario a usuarios legítimos que necesiten iniciar sesión en la consola de la computadora, los usuarios no autorizados podrían descargar y ejecutar software malicioso para elevar sus privilegios.

Impacto:
La eliminación del derecho de usuario "Permitir inicio de sesión a través de Remote Desktop Services" de otros grupos o cambios en la membresía de estos grupos predeterminados podrían limitar las habilidades de los usuarios que desempeñan roles administrativos específicos en su entorno. Debe confirmar que las actividades delegadas no se verán afectadas negativamente.

#SeRestorePrivilege
Descripción:
Este ajuste de política permite a los usuarios eludir los permisos de archivos y directorios para hacer una copia de seguridad del sistema. Este derecho de usuario se habilita solo cuando una aplicación (como NTBACKUP) intenta acceder a un archivo o directorio a través de la interfaz de programación de aplicaciones (API) de copia de seguridad del sistema de archivos NTFS. De lo contrario, se aplican los permisos de archivo y directorio asignados.

Justificación:
Los usuarios que pueden hacer copias de seguridad de datos desde una computadora podrían llevar los medios de copia de seguridad a una computadora no perteneciente al dominio en la que tienen privilegios administrativos y restaurar los datos. Podrían adquirir la propiedad de los archivos y ver cualquier dato no cifrado que esté contenido en el conjunto de copia de seguridad.

Impacto:
Los cambios en la membresía de los grupos que tienen el derecho de usuario "Hacer copia de seguridad de archivos y directorios" podrían limitar las habilidades de los usuarios que desempeñan roles administrativos específicos en su entorno. Debe confirmar que los administradores de copias de seguridad autorizados aún pueden realizar operaciones de copia de seguridad.

#SeSystemtimePrivilege
Descripción:
Este ajuste de política determina qué usuarios y grupos pueden cambiar la hora y la fecha en el reloj interno de las computadoras en su entorno. Los usuarios que tienen este derecho de usuario pueden afectar la apariencia de los registros de eventos. Cuando se cambia la configuración de tiempo de una computadora, los eventos registrados reflejan la nueva hora, no la hora real en que ocurrieron los eventos.

Justificación:
Los usuarios que pueden cambiar la hora en una computadora podrían causar varios problemas. Por ejemplo, los sellos de tiempo en las entradas del registro de eventos podrían ser inexactos, los sellos de tiempo en archivos y carpetas creados o modificados podrían ser incorrectos, y las computadoras que pertenecen a un dominio pueden no poder autenticarse o los usuarios que intentan iniciar sesión en el dominio desde ellas. Además, debido a que el protocolo de autenticación de Kerberos requiere que el solicitante y el autenticador tengan sus relojes sincronizados dentro de un período de desviación definido por el administrador, un atacante que cambie la hora de una computadora puede hacer que esa computadora no pueda obtener ni otorgar tickets de Kerberos.

Impacto:
No debería haber impacto, porque la sincronización de tiempo para la mayoría de las organizaciones debería estar completamente automatizada para todas las computadoras que pertenecen al dominio. Las computadoras que no pertenecen al dominio deben configurarse para sincronizarse con una fuente externa.

#SeTimeZonePrivilege
Descripción:
Este ajuste determina qué usuarios pueden cambiar la zona horaria de la computadora. Esta capacidad no representa un gran peligro para la computadora y puede ser útil para trabajadores móviles.

Justificación:
Cambiar la zona horaria representa una vulnerabilidad mínima porque la hora del sistema no se ve afectada. Este ajuste simplemente permite a los usuarios mostrar su zona horaria preferida mientras se sincronizan con Controladores de Dominio en diferentes zonas horarias.

Impacto:
Ninguno; este es el comportamiento predeterminado.

#SeCreatePagefilePrivilege
Descripción:
Este ajuste de política permite a los usuarios cambiar el tamaño del archivo de paginación. Al hacer que el archivo de paginación sea extremadamente grande o extremadamente pequeño, un atacante podría afectar fácilmente el rendimiento de una computadora comprometida.

Justificación:
Los usuarios que pueden cambiar el tamaño del archivo de paginación podrían hacerlo extremadamente pequeño o mover el archivo a un volumen de almacenamiento altamente fragmentado, lo que podría causar una reducción en el rendimiento de la computadora.

Impacto:
Ninguno; este es el comportamiento predeterminado.

#SeCreateTokenPrivilege
Descripción:
Este ajuste de política permite a un proceso crear un token de acceso, que puede proporcionar derechos elevados para acceder a datos sensibles.

Justificación:
Una cuenta de usuario que recibe este derecho tiene control total sobre el sistema y puede comprometerlo. Se recomienda encarecidamente no asignar ningún derecho de usuario a esta configuración. Los tokens de acceso se crean cuando los usuarios inician sesión en la computadora local o se conectan a una computadora remota a través de una red. Los usuarios con la capacidad de crear o modificar tokens pueden cambiar el nivel de acceso para cualquier cuenta actualmente conectada.

Impacto:
Ninguno; este es el comportamiento predeterminado.

#SeCreateGlobalPrivilege
Descripción:
Este ajuste de política determina si los usuarios pueden crear objetos globales disponibles para todas las sesiones. Los usuarios aún pueden crear objetos específicos de su sesión si no tienen este derecho de usuario. Los usuarios que pueden crear objetos globales podrían afectar los procesos que se ejecutan bajo sesiones de otros usuarios. Esta capacidad podría provocar una variedad de problemas, como fallos en las aplicaciones o corrupción de datos.

Justificación:
Los usuarios que pueden crear objetos globales podrían afectar los servicios de Windows y los procesos que se ejecutan bajo otras cuentas de usuario o del sistema. Esta capacidad podría llevar a una variedad de problemas, como fallos en las aplicaciones, corrupción de datos y elevación de privilegios.

Impacto:
Ninguno; este es el comportamiento predeterminado.

#SeCreatePermanentPrivilege
Descripción:
Este derecho de usuario es útil para los componentes del modo kernel que extienden el espacio de nombres del objeto. Sin embargo, los componentes que se ejecutan en modo kernel tienen este derecho de usuario inherente. Por lo tanto, generalmente no es necesario asignar específicamente este derecho de usuario.

Justificación:
Los usuarios que tienen el derecho de usuario para crear objetos compartidos permanentes podrían crear nuevos objetos compartidos y exponer datos sensibles a la red.

Impacto:
Ninguno; este es el comportamiento predeterminado.

#SeCreateSymbolicLinkPrivilege
Descripción:
Esta configuración de política determina qué usuarios pueden crear vínculos simbólicos. En Windows Vista, objetos del sistema de archivos NTFS existentes, como archivos y carpetas, pueden ser accedidos refiriéndose a un nuevo tipo de objeto del sistema de archivos llamado vínculo simbólico. Un vínculo simbólico es un puntero (similar a un acceso directo o archivo .lnk) a otro objeto del sistema de archivos, que puede ser un archivo, carpeta, acceso directo o otro vínculo simbólico. La diferencia entre un acceso directo y un vínculo simbólico es que un acceso directo solo funciona desde el shell de Windows. Para otros programas y aplicaciones, los accesos directos son solo otro archivo, mientras que con los vínculos simbólicos, el concepto de acceso directo se implementa como una característica del sistema de archivos NTFS. Los vínculos simbólicos pueden potencialmente exponer vulnerabilidades de seguridad en aplicaciones que no están diseñadas para usarlos. Por esta razón, el privilegio para crear vínculos simbólicos solo debe asignarse a usuarios de confianza. De forma predeterminada, solo los Administradores pueden crear vínculos simbólicos.

Justificación:
Los usuarios que tienen el derecho de usuario para crear vínculos simbólicos podrían exponer inadvertida o maliciosamente su sistema a ataques de vínculos simbólicos. Los ataques de vínculos simbólicos pueden ser utilizados para cambiar los permisos en un archivo, corromper datos, destruir datos o como un ataque de denegación de servicio.

Impacto:
En la mayoría de los casos no habrá impacto porque esta es la configuración predeterminada. Sin embargo, en estaciones de trabajo de Windows con la función Hyper-V instalada, este derecho de usuario también debe otorgarse al grupo especial NT VIRTUAL MACHINE\Virtual Machines; de lo contrario, no podrá crear nuevas máquinas virtuales.

#SeDebugPrivilege
Descripción:

Esta configuración de política determina qué cuentas de usuario tendrán el derecho de adjuntar un depurador a cualquier proceso o al kernel, lo que proporciona acceso completo a componentes sensibles y críticos del sistema operativo. Los desarrolladores que depuran sus propias aplicaciones no necesitan que se les asigne este derecho de usuario; sin embargo, los desarrolladores que depuran nuevos componentes del sistema lo necesitarán. El estado recomendado para esta configuración es: Administradores.

Nota:

Este derecho de usuario se considera un "privilegio sensible" para fines de auditoría.

Justificación:

El derecho de usuario Depurar programas puede ser explotado para capturar información sensible de la computadora desde la memoria del sistema, o para acceder y modificar estructuras del kernel o de la aplicación. Algunas herramientas de ataque explotan este derecho de usuario para extraer contraseñas con hash y otra información de seguridad privada, o para insertar código de rootkit. Por defecto, el derecho de usuario Depurar programas se asigna solo a los administradores, lo que ayuda a mitigar el riesgo de esta vulnerabilidad.

Impacto:

Si revoca este derecho de usuario, nadie podrá depurar programas. Sin embargo, circunstancias típicas rara vez requieren esta capacidad en computadoras de producción. Si surge un problema que requiere depurar una aplicación en un servidor de producción, puede mover el servidor a una Unidad Organizativa diferente temporalmente y asignar el derecho de usuario Depurar programas a una Política de Grupo separada para esa OU. La cuenta de servicio que se utiliza para el servicio de clúster necesita el derecho de usuario Depurar programas; si no lo tiene, el Clúster de Windows fallará. Las herramientas que se utilizan para administrar procesos no podrán afectar a procesos que no sean propiedad de la persona que ejecuta las herramientas. Por ejemplo, la herramienta Kill.exe del Kit de Recursos de Windows Server 2003 requiere este derecho de usuario para que los administradores puedan terminar procesos que no iniciaron.

#SeDenyNetworkLogonRight
Descripción:

Esta configuración de política prohíbe a los usuarios conectarse a una computadora desde la red, lo que les permitiría acceder y potencialmente modificar datos de forma remota. En entornos de alta seguridad, no debería haber necesidad de que los usuarios remotos accedan a datos en una computadora. En su lugar, el intercambio de archivos debería realizarse mediante el uso de servidores de red. Este derecho de usuario prevalece sobre el derecho de usuario Acceder a esta computadora desde la red si una cuenta está sujeta a ambas políticas. El estado recomendado para esta configuración es incluir: Invitados.

Justificación:

Los usuarios que pueden iniciar sesión en la computadora a través de la red pueden enumerar listas de nombres de cuentas, nombres de grupos y recursos compartidos. Los usuarios con permiso para acceder a carpetas y archivos compartidos pueden conectarse a través de la red y posiblemente ver o modificar datos.

Impacto:

Si configura el derecho de usuario Denegar el acceso a esta computadora desde la red para otros grupos, podría limitar las capacidades de los usuarios asignados a roles administrativos específicos en su entorno. Debería verificar que las tareas delegadas no se vean afectadas negativamente.

#SeDenyBatchLogonRight
Descripción:

Esta configuración de política determina qué cuentas no podrán iniciar sesión en la computadora como un trabajo por lotes. Un trabajo por lotes no es un archivo por lotes (.bat), sino más bien una facilidad de cola de lotes. Las cuentas que utilizan el Programador de tareas para programar trabajos necesitan este derecho de usuario. Este derecho de usuario prevalece sobre el derecho de usuario Iniciar sesión como trabajo por lotes, que podría utilizarse para permitir que las cuentas programen trabajos que consuman recursos del sistema de manera excesiva. Tal situación podría causar una condición de denegación de servicio (DoS). No asignar este derecho de usuario a las cuentas recomendadas puede ser un riesgo para la seguridad. El estado recomendado para esta configuración es incluir: Invitados.

Justificación:

Las cuentas que tienen el derecho de usuario Iniciar sesión como trabajo por lotes podrían utilizarse para programar trabajos que consuman recursos de la computadora de manera excesiva y causen una condición de denegación de servicio (DoS).

Impacto:

Si asigna el derecho de usuario Denegar el inicio de sesión como trabajo por lotes a otras cuentas, podría denegar a los usuarios asignados a roles administrativos específicos la capacidad de realizar sus actividades laborales requeridas. Debería confirmar que las tareas delegadas no se vean afectadas adversamente. Por ejemplo, si asigna este derecho de usuario a la cuenta IWAM_(NombreDeComputadora), el Punto de administración de MSM fallará. En una computadora recién instalada que ejecuta Windows Server 2003, esta cuenta no pertenece al grupo de Invitados, pero en una computadora que se actualizó desde Windows 2000, esta cuenta es miembro del grupo de Invitados. Por lo tanto, es importante que comprenda qué cuentas pertenecen a los grupos a los que asigna el derecho de usuario Denegar el inicio de sesión como trabajo por lotes.

#SeDenyServiceLogonRight
Descripción:

Esta configuración de seguridad determina qué cuentas de servicio se evitan de registrar un proceso como un servicio. Este derecho de usuario prevalece sobre el derecho de usuario Iniciar sesión como servicio si una cuenta está sujeta a ambas políticas. El estado recomendado para esta configuración es incluir: Invitados.

Nota:

Esta configuración de seguridad no se aplica a las cuentas System, Local Service o Network Service.

Justificación:

Las cuentas que pueden iniciar sesión como servicio podrían utilizarse para configurar y iniciar nuevos servicios no autorizados, como un keylogger u otro software malicioso. El beneficio de la contramedida especificada se ve algo reducido por el hecho de que solo los usuarios con privilegios administrativos pueden instalar y configurar servicios, y un atacante que ya haya obtenido ese nivel de acceso podría configurar el servicio para que se ejecute con la cuenta System.

Impacto:

Si asigna el derecho de usuario Denegar el inicio de sesión como servicio a cuentas específicas, los servicios pueden no poder iniciarse y podría resultar en una condición de denegación de servicio (DoS).

#SeDenyInteractiveLogonRight
Descripción:

Esta configuración de seguridad determina qué usuarios se evitan de iniciar sesión en la computadora. Esta configuración de política prevalece sobre la configuración de política Permitir el inicio de sesión local si una cuenta está sujeta a ambas políticas. El estado recomendado para esta configuración es incluir: Invitados.

Importante:

Si aplica esta política de seguridad al grupo de Todos, nadie podrá iniciar sesión localmente.

Justificación:

Cualquier cuenta con la capacidad de iniciar sesión localmente podría utilizarse para iniciar sesión en la consola de la computadora. Si este derecho de usuario no está restringido a usuarios legítimos que necesiten iniciar sesión en la consola de la computadora, los usuarios no autorizados podrían descargar y ejecutar software malicioso que eleve sus privilegios.

Impacto:

Si asigna el derecho de usuario Denegar el inicio de sesión local a cuentas adicionales, podría limitar las capacidades de los usuarios asignados a roles específicos en su entorno. Sin embargo, este derecho de usuario debe asignarse explícitamente a la cuenta ASPNET en computadoras que ejecuten IIS 6.0. Debería confirmar que las actividades delegadas no se vean afectadas adversamente.

#SeDenyRemoteInteractiveLogonRight
Descripción:

Esta configuración de política determina si los usuarios pueden iniciar sesión como clientes de Escritorio remoto. Este derecho de usuario prevalece sobre el derecho de usuario Permitir el inicio de sesión a través de los Servicios de Escritorio remoto si una cuenta está sujeta a ambas políticas. El estado recomendado para esta configuración es incluir: Invitados.

En todas las versiones de Windows anteriores a Windows 7, los Servicios de Escritorio remoto se conocían como Servicios de Terminal, por lo que debería sustituir el término anterior si lo está comparando con un sistema operativo más antiguo.

Si este derecho de usuario no se restringe a usuarios legítimos que necesiten iniciar sesión en la consola de la computadora, los usuarios no autorizados podrían descargar y ejecutar software malicioso que eleve sus privilegios.

Impacto:

Si asigna el derecho de usuario Denegar el inicio de sesión a través de los Servicios de Escritorio remoto a otros grupos, podría limitar las capacidades de los usuarios asignados a roles administrativos específicos en su entorno. Las cuentas que tienen este derecho de usuario no podrán conectarse a la computadora ni a través de los Servicios de Escritorio remoto ni de la Asistencia remota. Debería confirmar que las tareas delegadas no se vean afectadas negativamente.

#SeEnableDelegationPrivilege
Descripción:

Esta configuración de política permite a los usuarios cambiar el ajuste de Confianza para delegación en un objeto de computadora en Active Directory. El abuso de este privilegio podría permitir que usuarios no autorizados se hagan pasar por otros usuarios en la red. El estado recomendado para esta configuración es: Nadie.

Nota:

Este derecho de usuario se considera un "privilegio sensible" para fines de auditoría.

Justificación:

El mal uso del derecho de usuario Habilitar que las cuentas de equipo y usuario sean confiables para la delegación podría permitir que usuarios no autorizados se hagan pasar por otros usuarios en la red. Un atacante podría explotar este privilegio para obtener acceso a recursos de red y dificultar la determinación de lo que ha sucedido después de un incidente de seguridad.

Impacto:

Ninguno: este es el comportamiento predeterminado.

#SeRemoteShutdownPrivilege
Descripción:

Esta configuración de política permite a los usuarios apagar computadoras con Windows Vista o más recientes desde ubicaciones remotas en la red. Cualquier persona a la que se le haya asignado este derecho de usuario puede causar una condición de denegación de servicio (DoS), lo que haría que la computadora no esté disponible para atender solicitudes de usuario. Por lo tanto, se recomienda que solo se asignen este derecho de usuario a administradores altamente confiables. El estado recomendado para esta configuración es: Administradores.

Justificación:

Cualquier usuario que pueda apagar una computadora podría provocar una condición de denegación de servicio (DoS). Por lo tanto, este derecho de usuario debe estar restringido de manera estricta.

Impacto:

Si elimina el derecho de usuario Forzar apagado desde un sistema remoto del grupo de Operadores de servidor, podría limitar las capacidades de los usuarios asignados a roles administrativos específicos en su entorno. Debería confirmar que las actividades delegadas no se vean afectadas negativamente.

#SeAuditPrivilege
Descripción:

Esta configuración de política determina qué usuarios o procesos pueden generar registros de auditoría en el registro de seguridad. El estado recomendado para esta configuración es: LOCAL SERVICE, NETWORK SERVICE.

Nota:

Este derecho de usuario se considera un "privilegio sensible" para fines de auditoría.

Justificación:

Un atacante podría utilizar esta capacidad para crear un gran número de eventos auditados, lo que dificultaría que un administrador del sistema localizara cualquier actividad ilícita. Además, si el registro de eventos está configurado para sobrescribir eventos según sea necesario, cualquier evidencia de actividades no autorizadas podría ser sobrescrita por un gran número de eventos no relacionados.

Impacto:

En la mayoría de las computadoras, esta es la configuración predeterminada y no habrá impacto negativo. Sin embargo, si ha instalado un Servidor Web (IIS), necesitará permitir que el o los grupos de aplicaciones de IIS reciban este derecho de usuario.

#SeImpersonatePrivilege

Descripción:

Esta configuración de política permite a los programas que se ejecutan en nombre de un usuario suplantar a ese usuario (o a otra cuenta especificada) para que puedan actuar en nombre del usuario. Si este derecho de usuario es necesario para este tipo de suplantación, un usuario no autorizado no podrá convencer a un cliente para que se conecte, por ejemplo, mediante llamadas de procedimiento remoto (RPC) o tuberías nombradas, a un servicio que hayan creado para suplantar a ese cliente, lo que podría elevar los permisos del usuario no autorizado a niveles administrativos o del sistema. Los servicios que son iniciados por el Administrador de control de servicios tienen el grupo de Servicio incorporado agregado por defecto a sus tokens de acceso. Los servidores COM que son iniciados por la infraestructura COM y configurados para ejecutarse bajo una cuenta específica también tienen el grupo de Servicio agregado a sus tokens de acceso. Como resultado, estos procesos reciben este derecho de usuario cuando se inician. Además, un usuario puede suplantar un token de acceso si se cumplen cualquiera de las siguientes condiciones: • El token de acceso que se está suplantando es para este usuario. • El usuario, en esta sesión de inicio de sesión, se conectó a la red con credenciales explícitas para crear el token de acceso. • El nivel solicitado es menor que Suplantar, como Anónimo o Identificar. Un atacante con el derecho de usuario Suplantar un cliente después de la autenticación podría crear un servicio, engañar a un cliente para que se conecte al servicio y luego suplantar a ese cliente para elevar el nivel de acceso del atacante al del cliente. El estado recomendado para esta configuración es: Administradores, SERVICIO LOCAL, SERVICIO DE RED, SERVICIO.

Nota:

Este derecho de usuario se considera un "privilegio sensible" para fines de auditoría.

Justificación:

Un atacante con el derecho de usuario Suplantar un cliente después de la autenticación podría crear un servicio, engañar a un cliente para que se conecte al servicio y luego suplantar a ese cliente para elevar el nivel de acceso del atacante al del cliente.

Impacto:

En la mayoría de los casos, esta configuración no tendrá impacto. Si ha instalado un Servidor Web (IIS), también necesitará asignar el derecho de usuario a IIS_IUSRS.

#SeIncreaseBasePriorityPrivilege
Descripción:

Esta configuración de política determina si los usuarios pueden aumentar la clase de prioridad base de un proceso. (No es una operación privilegiada aumentar la prioridad relativa dentro de una clase de prioridad). Este derecho de usuario no es requerido por las herramientas administrativas suministradas con el sistema operativo pero podría ser requerido por herramientas de desarrollo de software. El estado recomendado para esta configuración es: Administradores, Grupo Window Manager\Window Manager.

Justificación:

Un usuario al que se le asigna este derecho de usuario podría aumentar la prioridad de planificación de un proceso a Tiempo Real, lo que dejaría poco tiempo de procesamiento para todos los demás procesos y podría llevar a una condición de denegación de servicio (DoS).

Impacto:

Ninguno - este es el comportamiento predeterminado.

#SeLoadDriverPrivilege
Descripción:

Esta configuración de política permite a los usuarios cargar dinámicamente un nuevo controlador de dispositivo en un sistema. Un atacante podría potencialmente utilizar esta capacidad para instalar código malicioso que aparenta ser un controlador de dispositivo. Este derecho de usuario es necesario para que los usuarios agreguen impresoras locales o controladores de impresoras en Windows Vista. El estado recomendado para esta configuración es: Administradores.

Los controladores de dispositivos se ejecutan como código altamente privilegiado. Un usuario que tenga el derecho de usuario Cargar y descargar controladores de dispositivo podría instalar inadvertidamente código malicioso que se hace pasar por un controlador de dispositivo. Los administradores deben tener un cuidado mayor e instalar solo controladores con firmas digitales verificadas.

Impacto:

Si elimina el derecho de usuario Cargar y descargar controladores de dispositivo del grupo Operadores de impresión u otras cuentas, podría limitar las capacidades de los usuarios asignados a roles administrativos específicos en su entorno. Debería asegurarse de que las tareas delegadas no se vean afectadas negativamente.

#SeLockMemoryPrivilege
Descripción:

Esta configuración de política permite que un proceso mantenga los datos en la memoria física, lo que evita que el sistema pagine los datos a la memoria virtual en el disco. Si se asigna este derecho de usuario, puede ocurrir una degradación significativa del rendimiento del sistema. El estado recomendado para esta configuración es: Nadie.

Justificación:

Los usuarios con el derecho de usuario Bloquear páginas en memoria podrían asignar memoria física a varios procesos, lo que podría dejar poco o ningún RAM para otros procesos y resultar en una condición de denegación de servicio (DoS).

Impacto:

Ninguno: este es el comportamiento predeterminado.

#SeBatchLogonRight
Descripción:

Esta configuración de política permite que las cuentas inicien sesión usando el servicio Programador de tareas. Dado que el Programador de tareas a menudo se utiliza con fines administrativos, puede ser necesario en entornos empresariales. Sin embargo, su uso debería restringirse en entornos de alta seguridad para prevenir el mal uso de los recursos del sistema o para evitar que los atacantes utilicen el derecho para ejecutar código malicioso después de obtener acceso de nivel de usuario a una computadora. El estado recomendado para esta configuración es: Administradores.

Justificación:

El derecho de usuario Iniciar sesión como tarea por lotes presenta una vulnerabilidad de bajo riesgo. Para la mayoría de las organizaciones, los valores predeterminados son suficientes.

Impacto:

Si configura la configuración de Iniciar sesión como tarea por lotes a través de las Directivas de grupo basadas en dominio, la computadora no podrá asignar el derecho de usuario a las cuentas que se utilizan para trabajos programados en el Programador de tareas. Si instala componentes opcionales como ASP.NET o IIS, es posible que necesite asignar este derecho de usuario a cuentas adicionales que sean requeridas por esos componentes. Por ejemplo, IIS requiere la asignación de este derecho de usuario al grupo IIS_WPG y a las cuentas IUSR_(NombreDeLaComputadora), ASPNET e IWAM_(NombreDeLaComputadora). Si este derecho de usuario no se asigna a este grupo y a estas cuentas, IIS no podrá ejecutar algunos objetos COM que son necesarios para su funcionamiento adecuado.

#SeBatchLogonRight
Descripción:

Esta configuración de política permite que las cuentas inicien servicios de red o registren un proceso como un servicio que se ejecuta en el sistema. Este derecho de usuario debería restringirse en cualquier computadora en un entorno de alta seguridad, pero debido a que muchas aplicaciones pueden requerir este privilegio, debería evaluarse y probarse cuidadosamente antes de configurarlo en un entorno empresarial. En computadoras basadas en Windows Vista (o más nuevas), ningún usuario o grupo tiene este privilegio de forma predeterminada. El estado recomendado para esta configuración es: Nadie o (cuando se instala la función Hyper-V) NT VIRTUAL MACHINE\Virtual Machines o (cuando se utiliza Windows Defender Application Guard, como en el perfil de Seguridad de Windows Next Generation) WDAGUtilityAccount.

Nota:

La función Hyper-V se introdujo por primera vez en estaciones de trabajo con la versión de 64 bits de Windows 8.0, por lo que la opción NT VIRTUAL MACHINE\Virtual Machines no se aplica a las versiones de Windows 7 (o anteriores) de Windows. Los sistemas operativos más antiguos solo deben configurarse para Nadie.

Nota #2:

La función Windows Defender Application Guard se introdujo por primera vez en estaciones de trabajo con la versión de 64 bits de Windows 10, por lo que la opción WDAGUtilityAccount no se aplica a versiones de 32 bits de Windows.

Justificación:

Iniciar sesión como servicio es un derecho de usuario poderoso porque permite que las cuentas inicien servicios de red o servicios que se ejecuten continuamente en una computadora, incluso cuando nadie está conectado a la consola. El riesgo se reduce por el hecho de que solo los usuarios con privilegios administrativos pueden instalar y configurar servicios. Un atacante que ya haya alcanzado ese nivel de acceso podría configurar el servicio para ejecutarse con la cuenta del Sistema Local.

Impacto:

Si ha instalado componentes opcionales como ASP.NET o IIS, es posible que necesite asignar el derecho de usuario Iniciar sesión como servicio a cuentas adicionales que sean requeridas por esos componentes. IIS requiere que este derecho de usuario se otorgue explícitamente a la cuenta de usuario ASPNET. En estaciones de trabajo de Windows con la función Hyper-V instalada, este derecho de usuario también debería otorgarse al grupo especial NT VIRTUAL MACHINE\Virtual Machines.

#SeSecurityPrivilege
Descripción:

Esta configuración de política determina qué usuarios pueden cambiar las opciones de auditoría para archivos y directorios y limpiar el registro de seguridad. El estado recomendado para esta configuración es: Administradores.

Nota:

Este derecho de usuario se considera un "privilegio sensible" para fines de auditoría.

Justificación:

La capacidad de gestionar el registro de eventos de seguridad es un derecho de usuario poderoso y debe ser custodiado de cerca. Cualquier persona con este derecho de usuario puede limpiar el registro de seguridad para borrar pruebas importantes de actividad no autorizada.

Impacto:

Ninguno - este es el comportamiento predeterminado.

#SeRelabelPrivilege
Descripción:

Este privilegio determina qué cuentas de usuario pueden modificar la etiqueta de integridad de objetos, como archivos, claves de registro o procesos propiedad de otros usuarios. Los procesos en ejecución bajo una cuenta de usuario pueden modificar la etiqueta de un objeto propiedad de ese usuario a un nivel más bajo sin este privilegio. El estado recomendado para esta configuración es: Nadie.

Justificación:

Al modificar la etiqueta de integridad de un objeto propiedad de otro usuario, un usuario malintencionado puede hacer que ejecuten código a un nivel de privilegio más alto de lo previsto.

Impacto:

Ninguno - este es el comportamiento predeterminado.

#SeSystemEnvironmentPrivilege
Descripción:

Esta configuración de política permite a los usuarios configurar las variables de entorno del sistema que afectan la configuración del hardware. Esta información se almacena típicamente en la Última Configuración Buena Conocida. La modificación de estos valores podría provocar una falla de hardware que resultaría en una condición de denegación de servicio. El estado recomendado para esta configuración es: Administradores.

Justificación:

Cualquier persona a la que se le asigne el derecho de usuario Modificar los valores del entorno del firmware podría configurar los ajustes de un componente de hardware para provocar su falla, lo que podría conducir a la corrupción de datos o una condición de denegación de servicio.

Impacto:

Ninguno - este es el comportamiento predeterminado.

#SeManageVolumePrivilege
Descripción:

Esta configuración de política permite a los usuarios administrar la configuración de volumen o disco del sistema, lo que podría permitir a un usuario eliminar un volumen y causar pérdida de datos, así como una condición de denegación de servicio. El estado recomendado para esta configuración es: Administradores.

Justificación:

Un usuario al que se le asigne el derecho de usuario Realizar tareas de mantenimiento de volumen podría eliminar un volumen, lo que podría resultar en la pérdida de datos o una condición de denegación de servicio.

Impacto:

Ninguno - este es el comportamiento predeterminado.

#SeProfileSingleProcessPrivilege
Descripción:

Esta configuración de política determina qué usuarios pueden utilizar herramientas para monitorear el rendimiento de procesos que no son del sistema. Por lo general, no es necesario configurar este derecho de usuario para usar el complemento Performance de la Consola de Administración de Microsoft (MMC). Sin embargo, se necesita este derecho de usuario si se configura Monitor del Sistema para recopilar datos utilizando Instrumentación de Administración de Windows (WMI). Restringir el derecho de usuario de Perfil de un solo proceso evita que los intrusos obtengan información adicional que podría usarse para lanzar un ataque contra el sistema. El estado recomendado para esta configuración es: Administradores.

Justificación:

El derecho de usuario de Perfil de un solo proceso presenta una vulnerabilidad moderada. Un atacante con este derecho de usuario podría monitorear el rendimiento de una computadora para ayudar a identificar procesos críticos que podrían querer atacar directamente. El atacante también podría determinar qué procesos se ejecutan en la computadora para identificar contramedidas que puedan necesitar evitar, como software antivirus, un sistema de detección de intrusos o qué otros usuarios están conectados a una computadora.

Impacto:

Ninguno - este es el comportamiento predeterminado.

#SeSystemProfilePrivilege
Descripción:

Esta configuración de política permite a los usuarios utilizar herramientas para ver el rendimiento de diferentes procesos del sistema, lo que podría ser utilizado de manera abusiva por atacantes para determinar los procesos activos de un sistema y proporcionar información sobre la superficie de ataque potencial de la computadora. El estado recomendado para esta configuración es: Administradores, NT SERVICE\WdiServiceHost.

Justificación:

El derecho de usuario Perfil de rendimiento del sistema representa una vulnerabilidad moderada. Los atacantes con este derecho de usuario podrían monitorear el rendimiento de una computadora para ayudar a identificar procesos críticos que podrían querer atacar directamente. Los atacantes también podrían determinar qué procesos están activos en la computadora para identificar contramedidas que puedan necesitar evitar, como software antivirus o un sistema de detección de intrusiones.

Impacto:

Ninguno - este es el comportamiento predeterminado.

#SeShutdownPrivilege
Descripción:

Esta configuración de política determina qué usuarios que hayan iniciado sesión localmente en las computadoras de su entorno pueden apagar el sistema operativo con el comando Apagar. El uso indebido de este derecho de usuario puede provocar una condición de denegación de servicio. El estado recomendado para esta configuración es: Administradores, Usuarios.

Justificación:

La capacidad de apagar una estación de trabajo debería estar disponible generalmente para administradores y usuarios autorizados de esa estación de trabajo, pero no permitida para invitados o usuarios no autorizados, para evitar un ataque de denegación de servicio.

Impacto:

El impacto de eliminar estos grupos predeterminados del derecho de usuario Apagar el sistema podría limitar las capacidades delegadas de los roles asignados en su entorno. Debe confirmar que las actividades delegadas no se verán afectadas adversamente.

#SeTakeOwnershipPrivilege
Descripción:

Esta configuración de política permite a los usuarios tomar posesión de archivos, carpetas, claves del registro, procesos o subprocesos. Este derecho de usuario omite cualquier permiso que esté vigente para proteger objetos y otorga la propiedad al usuario especificado. El estado recomendado para esta configuración es: Administradores.

Justificación:

Cualquier usuario con el derecho de usuario Tomar posesión de archivos u otros objetos puede tomar el control de cualquier objeto, independientemente de los permisos en ese objeto, y luego realizar los cambios que deseen en ese objeto. Tales cambios podrían resultar en la exposición de datos, corrupción de datos o una condición de denegación de servicio.

Impacto:

Ninguno - este es el comportamiento predeterminado.
